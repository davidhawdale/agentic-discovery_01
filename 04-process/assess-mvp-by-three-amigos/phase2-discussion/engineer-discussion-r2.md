# Feasibility Review — Round 2 Follow-Up

- **The three-trust-dimension framework is analytically correct but architecturally misleading for v1.** Data trust, competence trust, and agency trust are real and distinct — but they do not require three separate engineering subsystems. Data trust is handled by OAuth scoping and encryption-at-rest, which are table-stakes infrastructure decisions, not product features. Agency trust is handled by the approval gate, which is already in scope. Competence trust is handled by source attribution on memory-sourced outputs. The UX Researcher is right that these are different user concerns; I am saying they map onto infrastructure that is already planned, not onto new architecture. The risk is not missing subsystems — it is missing measurement.

- **Where I concede ground: the experiments must measure all three dimensions separately, even if the architecture does not separate them.** My Round 1 position implied that source attribution in the approval gate "covers" trust. That was imprecise. It covers the engineering deliverable, but it does not cover the experimental design. If Experiment 1 returns a low trust score, we need to know whether users distrust the data handling (fixable with better onboarding and transparency), the memory accuracy (fixable with retrieval improvements), or the action execution (fixable with approval gate UX). A single trust metric makes failure undiagnosable — the UX Researcher is right on this point, and the experiments should include distinct questions or signals for each dimension.

- **Source attribution is the only engineering change this implies for v1 scope.** Data trust requires standard security practices we would build anyway. Agency trust requires the approval gate we are already building. Competence trust requires source attribution, which I already promoted to v1 in my Round 1 recommendations. The three-trust framework does not add engineering scope — it adds experimental instrumentation. That is a design and measurement task, not a build task.

- **The daily briefing is where all three trust dimensions collide, and the UX Researcher is right that it is underdesigned.** I underweighted this in Round 1. The briefing is the first surface where the user encounters memory (competence trust), sees what data was accessed (data trust), and decides whether to act (agency trust). If the briefing design is vague, the experiments will test an undefined artifact and the trust measurements will be noise. I support the UX Researcher's call for a full briefing design spec before build — not because it changes the architecture, but because it defines what the architecture must render.

- **Remaining disagreement: the UX Researcher's framing risks scope inflation if taken as a build requirement rather than a measurement requirement.** "Three trust dimensions" can be read as "build three trust mechanisms." It should be read as "measure three trust outcomes." The engineering deliverables remain: OAuth + encryption (data trust), approval gate (agency trust), source attribution (competence trust). What changes is the experiment protocol, not the system design. If the panel agrees on this reading, we are aligned.
